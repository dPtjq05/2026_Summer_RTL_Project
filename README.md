### 2. Baud Rate Generator (클럭 분주기) 사양 정의

* **시스템 입력 클럭 ($f_{\text{clk}}$):** $12\text{ MHz}$
* **목표 보레이트 (Baud Rate):** $9,600\text{ bps}$
* **오버샘플링 배수:** $16\text{x}$ (노이즈 필터링 및 안정적인 비트 센터 저격을 위함)
* **목표 틱 주파수 ($f_{\text{tick}}$):** $9,600 \times 16 = 153,600\text{ Hz}$
* **카운터 분주비 (Division Ratio):**
  $$12,000,000\text{ Hz} \div 153,600\text{ Hz} = 78.125 \approx 78$$

* **분주기 동작 사양:**
  * 오차율 $0.15\%$ 미만의 고신뢰성 분주 계수 **`78`** 도출.
  * $0$부터 $77$까지 카운팅하기 위해 최소 **7비트 레지스터**(`reg [6:0]`) 설계 사양 확정.

## 📝 Day 2: UART RX 아키텍처 설계 및 샘플링 논리 정립

### 1. UART RX (수신기) 모듈 사양 및 기능 정의
선 한 가닥으로 들어오는 비동기 직렬(Serial) 데이터를 8비트 온전한 병렬(Parallel) 데이터 한 묶음으로 조립하는 디시리얼라이저(Deserializer)를 설계했습니다.
* **시리얼-투-패러렐 변환:** 1비트씩 순서대로 들어오는 `rx` 신호를 수집하여 8비트 레지스터 상자(`b_reg`)에 조립합니다.
* **1클럭 펄스 철학 반영:** 데이터 조립이 완벽히 끝나는 Stop 비트의 최종 시점에 정확히 **딱 1시스템 클럭 동안만 수신 완료 신호(`rx_done`)를 활성화**하여 상위 제어 시스템과의 동기화 신뢰성을 극대화했습니다.

### 2. 16x 오버샘플링 및 정중앙 저격 (Center Sampling) 메커니즘
송신측과 수신측의 미세한 클럭 오차 및 외부 노이즈로 인해 신호의 경계선(Edge)이 흔들리는 문제를 해결하기 위해 **16배수 오버샘플링 기술**을 도입했습니다. Baud Rate Generator가 쏴주는 1클럭 폭의 메트로놈 박수(`sampling_tick`)를 카운트하여 비트의 황금 입지(정중앙)를 정밀 타격합니다.
* **Start Bit 중심 잡기 (7번 카운트):** 평소 `1`이던 `rx` 선이 `0`으로 떨어지는 순간(Falling Edge) 진입합니다. 이후 틱을 7번 세어 **Start Bit의 딱 한가운데**에서 전압을 샘플링하여 진짜 데이터 신호인지 검증합니다. (노이즈일 경우 IDLE 복귀)
* **Data Bit 중심 저격 (16번 카운트):** Start Bit 중심 확보 이후부터는 비트 하나의 온전한 가로 폭인 **16번의 틱 간격(Center-to-Center)**마다 셔터를 눌러 `Data 0`부터 `Data 7`까지 노이즈 없는 깨끗한 알맹이 전압을 연속 수집합니다.

### 3. 제어(Control)와 연산(Data)의 분리: 2-Block FSM 도입 배경
FSM(상태 머신) 구조 없이 거대한 단일 마스터 카운터 하나만으로 수천 클럭을 세며 모든 조건문을 처리할 경우, 하드웨어 타이밍 관점에서 치명적인 문제가 발생함을 인지하고 이를 아키텍처 레벨에서 방어했습니다.
* **임계 경로(Critical Path) 및 전달 지연 방지:** 단일 카운터 방식을 쓰면 큰 비트수의 가산기와 비교기가 복잡하게 얽혀 조합회로 구름(Combinational Cloud)이 깊어집니다. 이는 12MHz 시스템 주기의 상한선인 **83.33ns 제한을 초과하여 셋업 타임 위반(Setup Time Violation)**을 일으킬 위험이 큽니다.
* **2-Block FSM의 이점:** 회로를 `IDLE`, `START`, `DATA`, `STOP`의 4가지 상태방으로 쪼개고, 각 방 안에서는 고작 0~15까지만 세는 **작은 4비트 로컬 카운터 (`s_reg`)만 동작**하게 만들었습니다. 비교 연산 회로가 극도로 얇고 단순해져서 지연 시간을 최소화하고 **타이밍 슬랙(Slack, 여유 시간)을 압도적으로 확보**했습니다. 또한 Vivado 도구의 FSM 자동 최적화 합성을 유도하여 가독성과 효율성을 모두 잡았습니다.

### 4. 입출력 포트 (I/O Interface) 명세
| 신호 이름 (Signal) | 방향 (I/O) | 비트 폭 (Width) | 역할 및 설명 |
| :--- | :---: | :---: | :--- |
| `clk` | Input | 1 bit | 12MHz 메인 시스템 클럭 (회로의 심장) |
| `rst_n` | Input | 1 bit | Active Low 비동기 리셋 신호 |
| `rx` | Input | 1 bit | 외부 물리 전선에서 들어오는 진짜 직렬 데이터 입력 라인 |
| `sampling_tick` | Input | 1 bit | 분주기(Baud Rate Gen)가 78클럭마다 발생시키는 1클럭 폭 제어 신호탄 |
| `rx_done` | Output | 1 bit | 수신 완료 알림 플래그 (정확히 1클럭 동안 `1` 유지) |
| `dout` | Output | 8 bit | 조립이 완료된 8비트 온전한 병렬 데이터 출력 버스 |

## 📅 Day 3: UART RX 코어 로직 구현 및 정석 FSM 아키텍처 최적화

오늘 진행한 핵심 작업은 `DATA` 및 `STOP` 상태방의 예외 처리를 완벽히 마스터하고, 단순 코딩(Code-and-Fix)이 아닌 **First-Time-Right(선 구조 기획, 후 번역)** 관점으로 디지털 회로의 타이밍 마진을 정밀하게 제어하는 아키텍처를 완성한 것입니다.

### 🚀 1. 주요 구현 성과

* **16배수 오버샘플링(Oversampling) 마스터 게이트 전략 수립**
  * 시스템 클럭(12MHz)과 메트로놈 박수(`sampling_tick`)의 시간 스케일 차이로 인해 발생할 수 있는 오작동을 차단하기 위해, FSM 내부 연산의 최외곽에 `if (sampling_tick)`을 문지기로 배치.
  * 78클럭 동안 문이 활짝 열려 노이즈에 노출되는 치명적인 버그를 막고, 오직 박수가 터지는 딱 1클럭의 찰나에만 정중앙(`7번째`, `15번째` 틱)을 저격하도록 설계.
* **업계 표준 Registered Output 구조 적용**
  * 출력 포트인 `dout`과 완료 깃발인 `rx_done`에 쌍둥이 구조(`current / next`) 레지스터를 적용하여 Glitch(노이즈 전압)를 차단.
  * `rx_done` 신호가 상위 시스템(FIFO, CPU 등)에서 안전하게 인지될 수 있도록 정확히 1 시스템 클럭 주기 동안만 튀어 오르는 깨끗한 펄스(Pulse) 형태로 제어.

### 🔍 2. Critical Troubleshooting (치명적 버그 해결 과정)

#### 🚨 문제 상황: 마지막 8번째 데이터 비트(Data 7)의 유실 현상
* `DATA` 상태방에서 비트 카운터가 마지막 인덱스인 `3'd7`에 도달했을 때, 15번째 틱 정중앙에서 `next_state = STOP`으로 분기하는 구조를 설계함.
* 그러나 시뮬레이션 가상 파형 검증 결과, 분기문(`if/else`)의 우선순위로 인해 정작 가장 중요한 마지막 8번째 데이터 비트 전압(`rx`)을 임시 상자에 밀어 넣는 우측 시프트 연산(`next_data = {rx, current_data[7:1]};`)이 차단된 채 `STOP` 방으로 튕겨 나가는 현상 발견.

#### 🛠️ 해결 아키텍처 (First-Time-Right)
* 임의로 코드를 수정하는 주먹구구식 접근을 지양하고, 하드웨어 데이터 경로(Datapath)를 기하학적으로 역추적함.
* 0~7번째 비트 모두 "15번째 틱 정중앙에서는 무조건 데이터를 시프트해야 한다"는 공통 분모를 도출, **시프트 연산 라인을 조건 분기문 직전(상단)으로 추출(Extract)**하여 연산이 물리적으로 차단되지 않도록 구조를 재정렬함.

```verilog
// 💡 정석으로 교정된 DATA 상태방 내부 타이밍 저격 구조
if (current_cnt_t == 4'd15) begin
    next_cnt_t = 0;
    next_data = {rx, current_data[7:1]}; // 몇 번째 비트든 무조건 선 수집!

    if (current_cnt_d == 3'd7) begin     // 그 후 마지막 비트였는지 검증
        next_state = STOP;
        next_cnt_d = 3'd0;
    end else begin
        next_cnt_d = current_cnt_d + 3'd1;
    end
end
